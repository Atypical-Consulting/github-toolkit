# Backlog Item Templates

Templates for the markdown files generated by the ghs-repo-scan skill.

For per-check fix suggestions, see the individual check files in `../../shared/checks/`.

## SUMMARY.md Template

```markdown
# Repo Scan: {owner}/{repo}

> Generated: {YYYY-MM-DD}
> Visibility: {Public|Private}
> Health Score: {score}/{max} ({percentage}%)
> Open Issues: {count}

## Health Score Breakdown

| Tier | Earned | Max | Bar |
|------|--------|-----|-----|
| Tier 1 — Required | {n} | {max} | {bar} |
| Tier 2 — Recommended | {n} | {max} | {bar} |
| Tier 3 — Nice to Have | {n} | {max} | {bar} |

## Health — Action Items

| # | Item | Tier | Points | Status |
|---|------|------|--------|--------|
| 1 | [README](health/tier-1--readme.md) | 1 | 4 | FAIL |
| 2 | ... | | | |

## Health — Passing Checks

- **LICENSE** — MIT License (1,080 bytes)
- ...

## Open Issues

| # | Issue | Labels | Age | Assignee | File |
|---|-------|--------|-----|----------|------|
| 42 | [Login page crashes](issues/issue-42--login-page-crashes.md) | bug | 12d | @user | issue-42--login-page-crashes.md |
| 108 | [Add dark mode](issues/issue-108--add-dark-mode.md) | enhancement | 45d | — | issue-108--add-dark-mode.md |
```

The progress bar uses `█` for filled and `░` for empty, 8 characters wide.

## Health Item Template

Each `[FAIL]` or `[WARN]` health check gets its own file under `health/`:

```markdown
# {Check Name}

| Field | Value |
|-------|-------|
| **Repository** | `{owner}/{repo}` |
| **Source** | Health Check |
| **Tier** | {1|2|3} — {Required|Recommended|Nice to Have} |
| **Points** | {points} |
| **Status** | {FAIL|WARN} |
| **Detected** | {YYYY-MM-DD} |

## What's Missing

{One sentence: what the check found.}

## Why It Matters

{Two to three sentences explaining the impact. Tailor to the repo context — e.g., mention "private repo" or "fork" if relevant.}

## How to Fix

### Quick Fix

```bash
{single command using the actual owner/repo}
```

### Full Solution

{Longer explanation with file templates if needed.}

## Acceptance Criteria

- [ ] {Concrete, verifiable condition matching the check's pass condition}
- [ ] {Second condition if needed}

## References

- {Link to relevant GitHub docs}
```

### Template Guidelines

- **What's Missing**: Keep it factual, one sentence. "No README.md file exists." not "You should have a README."
- **Why It Matters**: Explain the *impact*, not just restate the check. "Without a README, visitors have no context..." not "A README is required."
- **How to Fix**: The quick fix should be a single copy-pasteable command. The full solution provides context for more thoughtful implementation.
- **Acceptance Criteria**: Map directly to the check's pass condition. Use checkboxes so the user can track completion.

## Issue Item Template

Each open GitHub issue gets its own file under `issues/`:

```markdown
# {Issue Title}

| Field | Value |
|-------|-------|
| **Repository** | `{owner}/{repo}` |
| **Source** | Issue #{number} |
| **Labels** | {comma-separated labels, or "none"} |
| **Assignee** | {assignee login, or "unassigned"} |
| **Status** | OPEN |
| **Created** | {YYYY-MM-DD} |
| **Updated** | {YYYY-MM-DD} |

## Description

{Issue body, truncated to 500 characters if longer. If truncated, add "..." at the end.}

## References

- https://github.com/{owner}/{repo}/issues/{number}
```

### Issue Template Guidelines

- **Title**: Use the exact issue title from GitHub
- **Labels**: List all labels, comma-separated. If no labels, write "none"
- **Description**: Reproduce the issue body as-is (markdown preserved). Truncate at 500 characters for very long issues.
- Keep it simple — the backlog item is a local reference to the issue, not a replacement for it
