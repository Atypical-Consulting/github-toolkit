IMPORTANT notes on the rmcp macro API:
- The `#[tool_router]` attribute goes on an `impl GitHubAutomateMcpServer` block (NOT on `impl ServerHandler`).
- Each tool function returns `Result<CallToolResult, rmcp::Error>`. Use `CallToolResult::text(msg)` for success responses.
- For error responses in future real implementations, use `error_result()` from `error.rs` — it sets `is_error: Some(true)`.
- The `#[tool(aggr)]` attribute marks the aggregate parameter struct.
- Doc comments (`///`) on struct fields become `description` in the JSON Schema.
- If the rmcp 0.16 API uses different method signatures or macro names than shown above, consult the `examples/` directory in the rmcp crate source (especially `counter_stdio.rs` or similar). The research doc notes that older blog posts show `tool_box!` which was renamed to `#[tool_router]` in newer versions.

1. Rewrite `src-tauri/src/mcp/server.rs` to implement `ServerHandler` with the `#[tool_handler]` macro:

```rust
use anyhow::Result;
use rmcp::{
    ServerHandler,
    model::{ServerCapabilities, ServerInfo, Implementation, ProtocolVersion},
    tool_handler,
};

#[derive(Debug, Clone)]
pub struct GitHubAutomateMcpServer;

impl GitHubAutomateMcpServer {
    pub fn new() -> Result<Self> {
        Ok(Self)
    }
}

#[tool_handler]
impl ServerHandler for GitHubAutomateMcpServer {
    fn get_info(&self) -> ServerInfo {
        ServerInfo {
            protocol_version: ProtocolVersion::V_2024_11_05,
            capabilities: ServerCapabilities::builder()
                .enable_tools()
                .build(),
            server_info: Implementation {
                name: "github-automate-mcp".into(),
                version: env!("CARGO_PKG_VERSION").into(),
            },
            instructions: Some(
                "GitHub repository health diagnostics and backlog management. \
                 Use get_health_scores to check repo health, get_backlog_items \
                 to see pending work, and scan tools to run diagnostics.".into()
            ),
        }
    }
}
```

IMPORTANT: The `#[tool_handler]` macro on `impl ServerHandler` automatically wires up `list_tools` and `call_tool` dispatch to the `#[tool_router]` impl block in `tools.rs`. Both `impl` blocks target `GitHubAutomateMcpServer`.

If the rmcp 0.16 API requires different types (e.g., `ProtocolVersion` might be an enum variant like `Latest` or a string), adjust according to the actual rmcp type definitions. The research doc says `ProtocolVersion::V_2024_11_05` but verify against the crate's docs.rs page.

```rust
use anyhow::Result;
use github_automate_lib::mcp::server::GitHubAutomateMcpServer;
use rmcp::{transport::stdio, ServiceExt};

#[tokio::main]
async fn main() -> Result<()> {
    tracing_subscriber::fmt()
        .with_writer(std::io::stderr)
        .with_ansi(false)
        .with_env_filter(
            tracing_subscriber::EnvFilter::from_default_env()
                .add_directive(tracing::Level::INFO.into()),
        )
        .init();

    tracing::info!("github-automate-mcp starting");

    let server = GitHubAutomateMcpServer::new()?;
    let service = server.serve(stdio()).await?;
    service.waiting().await?;

    Ok(())
}
```

IMPORTANT: No `println!` anywhere. The `tracing::info!` calls go to stderr via the subscriber. The `serve(stdio())` call takes exclusive ownership of stdout for JSON-RPC framing.
